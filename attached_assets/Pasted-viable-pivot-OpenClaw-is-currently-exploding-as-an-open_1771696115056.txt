1. Concept Examination & Validation
The "Black Box" Calibration: Brilliant. By forcing users to use your hardcoded risk parameters (Growth, Capacity, Macro, Micro), you eliminate the risk of a user configuring things poorly and getting liquidated. You are selling "Managed Risk."

Max 2 API Keys: Smart limitation. It allows a developer to have one key for a staging/testing agent and one for their production agent, preventing spam and database bloat on your end.

System Notifications: Essential for IaaS (Infrastructure as a Service). If you change the underlying "Capacity" limits or Aave changes their interest rates, you need a way to broadcast this to the users without them having to check a Twitter feed.

The "Disconnect" Switch: Critical for trust in Web3. The user must always have a "panic button" to revoke your platform's access to their WBTC and pull their funds.

2. Technical Build Specs
To make this work, your architecture needs to split into three interconnected layers:

A. The Web3 Frontend (User Dashboard)
Auth: Implement "Sign-In with Ethereum" (EIP-4361). The user connects their wallet (e.g., MetaMask) and signs a message to prove ownership. No emails, no passwords.

Key Management UI: A simple panel where the user clicks "Generate API Key". The raw key is shown only once. The UI tracks how many active keys they have (enforcing the limit of 2).

Notification Inbox: A simple feed in the dashboard fetching broadcast messages from your database.

The "Eject" Button: A one-click function that revokes smart contract allowances and withdraws their WBTC, subsequently disabling their API keys.

B. The FastAPI Backend (The Bridge)
Database (SQLite or PostgreSQL): You need three basic tables:

Users: wallet_address (Primary Key).

API_Keys: key_hash (Primary Key), wallet_address (Foreign Key), status (active/revoked), created_at.

Notifications: message, date_sent.

Key Security: You never save the raw API key in the database. You hash it (using bcrypt or hashlib) just like a password. When the OpenClaw agent sends the key in the header, your backend hashes it and compares it to the database.

Rate Limiting: Use a library like slowapi or built-in token-bucket logic to prevent an AI agent from spamming your /borrow endpoint in an infinite loop.

C. The DeFi Engine (Multi-Tenant Isolation)
Your current bot logic likely assumes one wallet (yours). To become an API, the Python bot must become Multi-Tenant.

When an API key hits the /borrow endpoint, the backend must map that key to the specific user's wallet_address and execute the Capacity/Growth checks only against that specific user's isolated collateral vault, not a global pool.
