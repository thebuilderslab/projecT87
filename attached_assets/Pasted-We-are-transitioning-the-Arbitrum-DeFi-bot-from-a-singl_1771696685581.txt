We are transitioning the Arbitrum DeFi bot from a single-tenant architecture to a multi-tenant execution engine. We will start by updating the PostgreSQL schema to support API keys/notifications, and then rewrite the main bot loop to process all active delegated wallets concurrently.

TASK 1: Database Expansion (Phase 1)
File: db.py (or your database initialization script)

Action: Add two new tables to the PostgreSQL schema.

api_keys table:

id (SERIAL PRIMARY KEY)

user_id (INTEGER REFERENCES users(id))

key_hash (VARCHAR(128) NOT NULL UNIQUE)

key_prefix (VARCHAR(8)) - For UI display

status (VARCHAR(20) DEFAULT 'active')

created_at (TIMESTAMPTZ DEFAULT NOW())

notifications table:

id (SERIAL PRIMARY KEY)

wallet_address (VARCHAR(42) REFERENCES users(wallet_address))

message (TEXT NOT NULL)

created_at (TIMESTAMPTZ DEFAULT NOW())

Constraint: Create helper functions to generate, hash, and validate these keys. Add logic enforcing a maximum of 2 active keys per user_id.

TASK 2: Refactoring the Monolith (Phase 2)
File: arbitrum_testnet_agent.py (and associated strategy modules)

Action: Remove all instances of the hardcoded user wallet (0x5B82...) from the class initialization and method signatures.

New Flow:

_execute_liability_short_entry, check_capacity_opportunities, run_nurse_mode, and all other core functions must now accept a user_wallet_address parameter.

The bot will use its own Private Key (the Bot Operator) to sign transactions, but it will use the user_wallet_address to read the specific Aave health factor and execute the delegated_borrow functions.

TASK 3: The Concurrent Multi-Tenant Loop
File: main.py

Action: Rewrite the main_loop() to be multi-tenant.

Logic Requirements:

Query the managed_wallets table for all users where delegation_status = 'active'.

Do NOT process these users sequentially with a standard for loop (this creates a liquidation risk bottleneck for users at the end of the list).

Implement asyncio.gather() or a ThreadPoolExecutor to run the check_collateral_velocity and check_macro_micro_triggers concurrently across all active users.

Defense First: If multiple users trigger a Macro short simultaneously, the bot must queue and execute the delegated transactions rapidly.

TASK 4: The Operator Gas Reimbursement (Nurse Update)
Action: Update the run_nurse_mode(user_wallet_address) function.

Logic: Because the Bot Operator wallet pays the gas fees for these multi-tenant transactions, the Nurse function must sweep a portion of the "USDC Tax" from the user's position directly back to the Bot Operator's wallet to cover network costs.

Execute this refactor and confirm when the database is updated and the concurrent loop is in place.
