I want to refine the approvals and onboarding UX so that strategies are safe but don’t nag users on every run. Please implement the following behavior and code changes.

High-level requirement
Use on‑demand, one‑time approvals per token during onboarding and first use.

A user should not have to approve a token every time a strategy runs, only the first time that token is needed (unless they revoke/reduce it later).

The strategy engine must enforce that all required tokens are approved and safe before any strategy sequence executes.

1. Approval model and onboarding flow
1.1. Per‑token, persistent approvals
For each supported token T ∈ {WBTC, WETH, USDC, USDT, DAI}:

Maintain the invariant:

allowance(user, DelegationManager, T) >= required_threshold(T)

allowance(DelegationManager, AavePool, T) is set via a one‑time owner / deployment script (DM→Pool approval), not by end users.

Implement on‑demand approval:

The first time a user enables or runs a strategy that might supply/borrow token T, check allowance.

If insufficient, prompt exactly one approval tx for T → DelegationManager.

After it confirms, future runs that use T should not re‑prompt unless:

The allowance is now 0 or below the required threshold, or

We intentionally raise required_threshold(T).

1.2. Onboarding checklist
When a user connects and enables “full automation”:

Do not automatically ask for all 5 approvals.

Instead, present an “Advanced setup” checklist that:

Shows tokens that will be used by current strategies.

Offers a “Pre‑approve recommended tokens” option (optional UX enhancement), but the underlying logic still works if they skip it.

Internally, your strategy engine should still rely on on‑demand checks; the onboarding UI is just a convenience.

2. Strategy engine readiness and approvals
Update the readiness + strategy logic as follows:

In validate_full_automation_ready (and/or the equivalent readiness function):

For each token actually used by the active strategies for this wallet:

Check allowance(user, DM, token) >= required_threshold(token).

If any is insufficient:

Mark readiness as NOT READY – MISSING_ALLOWANCE(token).

Log a clear structured message (wallet, token, required vs current).

Ensure the frontend can surface this to the user (“Approve TOKEN to enable strategies”).

Before run_delegated_strategy executes any pipeline:

Call a helper like ensure_user_token_approvals(wallet) that:

Uses the above checks.

Does not auto‑send approvals, only determines if they are sufficient.

If approvals are missing:

Do not run the strategy.

Return a structured result that the API / frontend can interpret to show an “approval required” state.

Confirm that:

DM→Aave Pool approvals are set once at deployment or via an owner script (for all tokens your strategies use).

Strategies never rely on per‑execution DM→Pool approvals.

3. Frontend / API behavior
Developer portal / dashboard

In the wallet/automation settings UI:

Show, for each token:

Approved / Not approved status for user → DM.

Provide an “Approve” button per token that:

Calls a backend that returns the spender address and amount, or

Directly builds the ERC‑20 approve tx in the frontend to DelegationManager.

On first strategy use

If readiness fails with MISSING_ALLOWANCE(token):

Surface a clear message: “You need to approve TOKEN to the automation contract before strategies can run.”

Show the appropriate approval button.

No repeated prompts

Do not re‑prompt for a token that is already approved above threshold.

Only re‑prompt if:

Allowance dropped below threshold (due to user revoking/reducing), or

You explicitly increased required_threshold(token) and detect that the old allowance is insufficient.

Confirm no “missing allowance” state and no additional approval prompts.
